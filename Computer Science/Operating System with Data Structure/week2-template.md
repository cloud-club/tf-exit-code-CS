## [COMMON-041] 프로세스 주소 공간이 어떻게 구성되는지 설명해 주세요.

답변:
프로세스의 주소공간은 Stack, Heap, Data, Code 영역으로 구분됩니다.
Code 영역은 프로그램이 실행될 수 있도록 CPU가 해석 가능한 기계어 코드가 저장되어있는 곳을 의미합니다. 실행중인 프로그램의 코드가 들어가는 영역이므로, 수정되면 안되기에 ReadOnly 상태로 저장되어 있습니다.
Data 영역은 프로그램에서 사용되는 전역변수나 static 변수 등의 데이터를 저장하는 영역입니다. 프로그램이 컴파일된 후 data 영역을 참조하게 되며, 프로그램 시작 시 할당되어 종료 시점에 내부 데이터가 소멸되는 위치입니다.
Stack 영역은 함수의 호출과 관계되는 지역변수 또는 함수호출 시 전달된 매개변수들이 저장되는 영역으로, 메모리의 높은 곳부터 낮은 방향으로 데이터가 할당됩니다.
Heap 영역은 런타임에 크기가 결정되는 영역으로, 사용자에 의해 공간이 동적으로 할당 및 해제되는 곳을 의미합니다. 메모리의 낮은주소부터 높은 주소 방향으로 할당되며, 이를 통해 Heap과 Stack은 서로 마주보는 형태로 데이터가 쌓이게 됩니다.

## [COMMON-042] Code, Data, Heap, Stack 영역의 역할을 설명해 주세요.

답변:
Code 영역은 CPU에서 프로그램을 실행하기 위해 호출하는 기계명령들을 저장해주는 역할을 담당하며, Data 영역은 전역변수 또는 static 변수가 저장되는 공간입니다.
Heap 영역은 사용자에 의해 동적으로 할당되는 값들을 저장하는 역할을 담당하며, 대표적으로 참조형 데이터 등이 해당 영역에 저장됩니다.
Stack 영역은 프로세스의 실행 간 사용되는 지역변수 또는 매개변수를 저장하는 역할을 담당하며, 각 스레드가 개별적인 영역을 가지는 메모리 공간입니다.

## [COMMON-043] Stack과 Heap의 차이에 대해 설명해 주세요.

답변:
Stack과 Heap은 저장되는 데이터의 유형, 크기의 결정시점, 프로세스 별 영역의 존재개수 등에서 차이가 있습니다.
Stack의 경우 지역변수나 매개변수 등이 저장되고 크기 또한 프로세스가 생성되는 시점에 결정되며, 프로세스가 가지는 스레드의 개수에 따라 Stack의 개수가 결정됩니다.
Heap의 경우 사용자가 동적으로 할당한 데이터들이 저장되고 크기 또한 프로세스가 실행되는 런타임에 결정됩니다. 또한 Heap의 경우 각 프로세스 당 스레드의 개수에 무관하게 항상 1개의 영역만 존재합니다.

## [COMMON-044] Stack Overflow가 발생하는 이유를 설명해 주세요.

답변:

## [COMMON-045] Memory Leak이 무엇이고, 어떤 상황에서 발생할 수 있는지 설명해 주세요.

답변:
Memory Leak은 동적할당한 메모리공간을 해제하지 않아 가용메모리 공간이 부족해지는 현상을 의미합니다. 
Java의 경우, Memory Leak은 Garbage Collector에 의해 더이상 사용되지 않는 객체들이 정리되지 않는 현상을 의미하며, Map이나 Stream 객체의 올바르지 않은 사용으로 인해 보통 이 문제가 발생할 수 있습니다.
대표적으로는 무의미한 Wrapper 클래스 사용으로 인한 누수, 캐시 미삭제로 인한 메모리 누수, 스트림 미종료로 인한 메모리 누수, 맵키 재정의 누락 또는 키의 지속적 변화로 인한 메모리 누수 등이 있습니다.

## [COMMON-047] 페이징이 무엇이고, 페이지와 프레임의 차이를 설명해 주세요.

답변:
페이징은 가상 메모리 기술에서 실제 메모리 공간에 올라갈 메모리 공간단위를 관리하게 위해 사용되는 기술로, 페이지와 프레임 단위를 사용해 가상메모리 공간과 물리메모리 공간을 관리하는 기법입니다.

페이지와 프레임은 '어떤 메모리의 공간을 나누는 단위인가'라는 점에서 차이가 있습니다.
페이지는 페이징에서 '가상메모리'의 공간을 관리하기 위해 사용되는 단위를 의미하며,
프레임은 페이징에서 '물리메모리'의 공간을 관리하기 위해 사용하는 단위를 의미합니다.

위 2개의 단위를 통해, 페이징에서는 Page Fault의 감지를 통해 해당 데이터가 실제로 저장되어있는 디스크 공간으로부터 데이터를 가져와 메모리에 적재하고, 이를 통해 메모리를 보다 효율적으로 사용할 수 있게 해줍니다.

## [COMMON-048] Page Fault가 발생했을 때 처리 과정을 설명해 주세요.

답변:
Page Fault란 프로세스가 특정 메모리영역에 저장되어있는 데이터를 가져오려고 시도할 때, 물리메모리에 해당 데이터가 존재하지 않는 상황을 의미합니다.

Page Fault는 보통 CPU 내에 있는 메모리 관리 유닛 MMU가 Page Table 내의 실제 메모리주소를 확인했을 때 접근 가능한 유효한 메모리 영역을 확인하지 못할 때 발생합니다.
해당 문제는 OS의 커널모드에서 Page Fault Handler가 해결을 시도하는데, 크게 요청된 주소의 유효성 판단, 물리메모리 프레임 확보, 페이지 적재, 페이지테이블 갱신을 통해 문제를 해결합니다.
이후 OS에서 Page Fault가 발생했던 코드를 다시 실행하도록 함으로서, 실제 프로그램의 실행에 문제가 없도록 메모리서비스를 제공합니다.

## [COMMON-049] TLB가 무엇이고, TLB를 사용하면 왜 주소 변환이 빨라지는지 설명해 주세요.

답변:
TLB는 Translation Lookaside Buffer의 약자로, 가상메모리를 사용하기 위해 CPU가 메인메모리 대신 사용하는 버퍼입니다.
기존 페이징 사용 시, Page Table은 물리메모리에 적재되어야 했고, CPU가 가상메모리에 접근하기 위해서는 반드시 Page Table이 저장되어있는 물리메모리 위치에 최소 1번은 접근해야 했습니다.
CPU 내부에 비해 상대적으로 메모리의 접근속도는 느리기 때문에, 해당 접근과정을 개선하고자 TLB라는 장치를 두어 CPU 내에서 자주 접근하는 물리메모리 주소를 캐싱하고 자주 사용하는 가상메모리의 주소는 물리메모리의 Page Table 확인 없이 물리메모리의 실제 데이터 저장위치로 접근은 시도합니다.
따라서 Page Table이 저장되어있는 물리메모리 영역에 대한 왕복접근시간이 사라지므로, TLB를 사용 시 물리메모리에 대한 접근속도가 빨라집니다.

## [COMMON-050] 내부 단편화와 외부 단편화의 차이에 대해 설명해 주세요.

답변:
내부 단편화는 메모리를 관리하기 위해 나누어둔 공간단위를 프로세스가 사용하는 메모리 공간보다 커 낭비되는 문제를 의미합니다.
외부 단편화는 프로세스의 실행/종료가 반복되며 중간중간 생긴 미사용 메모리 영역들로 인해, 잔여 메모리 양의 총합은 충분하지만 실제로는 프로세스를 실행할 수 없는 문제를 의미합니다.

추가답변:

이를 해결하기 위해 페이징, 세그멘테이션, 메모리 풀 등의 기법을 사용할 수 있습니다.
페이징은 가상메모리를 통해 물리메모리공간보다 더 큰 메모리를 프로세스가 사용할 수 있도록 해주는 기법으로, 연속적이지 않은 공간에도 프로세스 메모리를 할당할 수 있기 때문에 외부 단편화 문제를 해결할 수 있습니다.
세그멘테이션은 가상메모리를 실제 프로세스가 필요로 하는 크기로 나누어 메모리를 할당하는 기법으로, 프로세스가 필요로 하는 딱 맞는 용량의 메모리를 제공하기 때문에 내부 단편화 문제를 해결할 수 있습니다.
메모리 풀은 필요한 메모리공간의 개수와 크기를 사용자가 직접 지정해 미리 할당받아두고, 이를 프로세스가 사용/반납하는 방식으로, 미리 공간을 할당해두고 사용/반납하기에 메모리 할당/해제로 인한 외부단편화가 발생하지 않으며, 필요한 크기만큼만 할당해두기 때문에 내부단편화도 발생하지 않습니다.

답변:

## [COMMON-053] Critical Section이 무엇이고, 임계 구역 문제를 해결하기 위한 조건을 설명해 주세요.

답변:
Race Condition 문제를 해결하기 위해서는 공유자원에 대한 접근을 제한하면 되므로 여러 스레드가 동시에 접근하지 못하도록 공유자원에 접근하는 코드로의 진입/진출을 제어함으로서 해결할 수 있고, 이때 공유자원에 접근하는 코드를 Critical Section이라 합니다.

Critical Section을 두었을 때, 크게 상호배제, 진행, 한정대기 조건이 충족되어야 합니다.
상호배제는 Critical Section에 접근하는 스레드의 개수가 반드시 1개여야 한다는 조건입니다.
진행은 한 프로세스가 다른 프로세스의 실행을 방해해서는 안된다는 조건입니다.
마지막 한정대기는 어떤 프로세스도 Critical Section에 접근하지 못해 무한히 대기해서는 안된다는 조건입니다.
