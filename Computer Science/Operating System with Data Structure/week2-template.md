## [COMMON-041] 프로세스 주소 공간이 어떻게 구성되는지 설명해 주세요.

답변:
프로세스의 주소공간은 대표적으로 Stack, Heap, Data, Code 영역으로 구분됩니다.

Code 영역은 프로그램이 실행될 수 있도록 CPU가 해석 가능한 기계어 코드가 저장되어 있는 곳을 의미합니다. 실행 중인 프로그램의 코드가 들어가는 영역이므로 일반적으로 읽기 및 실행 권한이 설정되고, 쓰기는 제한됩니다.
Data 영역은 프로그램에서 사용되는 전역변수나 static 변수 등의 데이터를 저장하는 영역입니다. 프로그램 시작 시 할당되어 프로그램이 종료될 때까지 유지됩니다. 초기화된 변수는 Data 영역에, 초기화되지 않은 변수는 BSS 영역에 저장됩니다.
Stack 영역은 함수 호출과 관계되는 지역변수, 매개변수, 반환 주소 등이 저장되는 영역입니다. 일반적으로 메모리의 높은 주소에서 낮은 주소 방향으로 증가하며, 각 스레드는 독립적인 Stack 영역을 가집니다.
Heap 영역은 런타임에 사용자 또는 프로그램에 의해 공간이 동적으로 할당되고 해제되는 영역입니다. 일반적으로 낮은 주소에서 높은 주소 방향으로 증가합니다.

## [COMMON-042] Code, Data, Heap, Stack 영역의 역할을 설명해 주세요.

답변:
Code 영역은 CPU가 프로그램을 실행하기 위해 사용하는 기계어 명령들을 저장하는 역할을 담당하며, Data 영역은 전역변수 또는 static 변수가 저장되는 공간입니다.
Heap 영역은 사용자 또는 프로그램에 의해 동적으로 할당되는 데이터를 저장하는 역할을 담당하며, Java와 같은 언어에서는 일반적으로 객체가 해당 영역에 저장됩니다. 객체를 가리키는 참조 변수 자체는 Stack 등에 저장될 수 있습니다.
Stack 영역은 프로세스 실행 중 사용되는 지역변수, 매개변수, 반환 주소 등을 저장하는 역할을 담당하며, 각 스레드가 개별적인 영역을 가지는 메모리 공간입니다.

## [COMMON-043] Stack과 Heap의 차이에 대해 설명해 주세요.

답변:
Stack과 Heap은 저장되는 데이터의 유형, 메모리 관리 방식, 프로세스별 영역의 존재 개수 등에서 차이가 있습니다.

Stack의 경우 지역변수, 매개변수, 반환 주소 등이 저장되고 함수 호출과 종료에 따라 자동으로 할당 및 해제됩니다. Stack은 각 스레드가 생성될 때 스레드별로 할당되므로, 프로세스가 가지는 스레드의 개수에 따라 Stack 영역의 개수가 결정됩니다.
Heap의 경우 사용자가 동적으로 할당한 데이터들이 저장되고, 프로그램 실행 중 필요에 따라 공간이 할당 및 해제됩니다. 또한 Heap은 프로세스 내부의 여러 스레드가 공유하는 메모리 영역입니다.

## [COMMON-044] Stack Overflow가 발생하는 이유를 설명해 주세요.

답변:
Stack Overflow는 Stack이 저장할 수 있는 메모리공간의 범위를 초과하는 만큼의 데이터가 저장될 떄 발생합니다.
Stack은 지역변수나 함수에 전달되는 매개변수들이 저장된 위치라는 점에서, 일반적으로 무한하게 재귀호출이 발생하거나 지역변수는 과도하게 크게 설정하는 경우 해당 에러가 발생할 수 있습니다.

## [COMMON-045] Memory Leak이 무엇이고, 어떤 상황에서 발생할 수 있는지 설명해 주세요.

답변:

Memory Leak은 할당한 메모리공간이 더 이상 필요하지 않음에도 적절히 해제되지 않거나 계속 참조되어, 가용 메모리 공간이 점차 부족해지는 현상을 의미합니다.

C나 C++에서는 동적으로 할당한 메모리를 해제하지 않거나, 해당 메모리를 가리키는 포인터를 잃어버린 경우에 발생할 수 있습니다.
Java의 경우 Garbage Collector가 존재하지만, 더 이상 필요하지 않은 객체가 static 변수, Collection, Cache, Listener 등에 의해 계속 참조되고 있다면 Garbage Collector의 정리 대상이 되지 않아 Memory Leak이 발생할 수 있습니다.

대표적으로는 크기나 만료 정책이 없는 Cache, Collection에서 사용하지 않는 객체를 제거하지 않는 경우, Listener나 Callback을 해제하지 않는 경우, ThreadLocal의 값을 제거하지 않는 경우 등이 있습니다.

Stream이나 File을 종료하지 않는 문제는 일반적으로 메모리 누수라기보다는 파일 디스크립터나 소켓 등의 자원 누수에 해당합니다.

## [COMMON-046] 가상 메모리가 무엇이고, 왜 필요한지 설명해 주세요.

답변:
가상 메모리란 프로세스가 실제 물리 메모리의 주소를 직접 사용하는 대신, 각 프로세스에 독립적인 가상 주소 공간을 제공하는 메모리 관리 기법입니다.

운영체제는 페이징이나 세그멘테이션 등의 기법을 통해 가상 주소를 물리 주소로 변환하며, 실제로 필요한 메모리 영역만 물리 메모리에 할당할 수 있습니다. 물리 메모리가 부족한 경우에는 일부 페이지가 디스크의 Swap 영역으로 이동할 수도 있습니다.

가상 메모리를 사용하면 프로세스마다 독립된 주소 공간을 제공할 수 있고, 프로세스 간 메모리 접근을 보호할 수 있으며, 물리 메모리를 효율적으로 사용할 수 있습니다. 또한 프로세스가 물리 메모리 크기보다 큰 주소 공간을 사용할 수 있도록 지원할 수 있습니다.

## [COMMON-047] 페이징이 무엇이고, 페이지와 프레임의 차이를 설명해 주세요.

답변:
페이징은 가상 메모리 기술에서 실제 메모리 공간에 올라갈 메모리 공간단위를 관리하게 위해 사용되는 기술로, 페이지와 프레임 단위를 사용해 가상메모리 공간과 물리메모리 공간을 관리하는 기법입니다.

페이지와 프레임은 '어떤 메모리의 공간을 나누는 단위인가'라는 점에서 차이가 있습니다.
페이지는 페이징에서 '가상메모리'의 공간을 관리하기 위해 사용되는 단위를 의미하며,
프레임은 페이징에서 '물리메모리'의 공간을 관리하기 위해 사용하는 단위를 의미합니다.

위 2개의 단위를 통해, 페이징에서는 Page Fault의 감지를 통해 해당 데이터가 실제로 저장되어있는 디스크 공간으로부터 데이터를 가져와 메모리에 적재하고, 이를 통해 메모리를 보다 효율적으로 사용할 수 있게 해줍니다.

## [COMMON-048] Page Fault가 발생했을 때 처리 과정을 설명해 주세요.

답변:
Page Fault란 프로세스가 특정 가상 메모리 주소에 접근하려고 할 때, 해당 페이지가 현재 물리 메모리에 존재하지 않거나 접근 권한을 위반한 경우 CPU가 발생시키는 예외를 의미합니다.

Page Fault는 보통 CPU 내부의 메모리 관리 유닛인 MMU가 Page Table을 확인했을 때, Page Table Entry의 Present Bit가 설정되어 있지 않거나 접근 권한에 위배되는 경우 발생합니다.
Page Fault가 발생하면 운영체제의 Page Fault Handler가 요청된 주소가 프로세스에 허용된 유효한 주소인지 먼저 판단합니다.
유효한 주소이지만 페이지가 물리 메모리에 없다면 물리 메모리의 Frame을 확보하고, 필요한 페이지를 디스크나 실행 파일로부터 적재한 후 Page Table과 TLB를 갱신합니다. 이후 Page Fault가 발생했던 명령어를 다시 실행합니다.
반대로 프로세스가 접근할 수 없는 잘못된 주소이거나 권한을 위반한 접근이라면 운영체제는 해당 프로세스에 예외나 Signal을 전달하며, 경우에 따라 프로세스를 종료합니다.

## [COMMON-049] TLB가 무엇이고, TLB를 사용하면 왜 주소 변환이 빨라지는지 설명해 주세요.

답변:
TLB는 Translation Lookaside Buffer의 약자로, CPU 내부에서 최근 사용된 가상 페이지 번호와 물리 프레임 번호의 변환 결과를 저장하는 캐시입니다.

기존 페이징 사용 시 Page Table은 일반적으로 물리 메모리에 저장되므로, CPU가 가상 주소를 물리 주소로 변환하기 위해서는 Page Table이 저장된 메모리에 접근해야 합니다. 특히 다단계 Page Table을 사용하는 경우 하나의 주소를 변환하기 위해 Page Table에 여러 번 접근해야 할 수 있습니다.

CPU는 이러한 주소 변환 시간을 줄이기 위해 TLB에 최근 사용한 주소 변환 결과를 저장합니다. CPU가 가상 주소에 접근했을 때 해당 변환 정보가 TLB에 존재하는 TLB Hit가 발생하면, Page Table을 다시 탐색하지 않고 즉시 물리 프레임 번호를 얻을 수 있습니다. 이를 통해 Page Table 탐색에 필요한 추가적인 메모리 접근을 줄여 전체 주소 변환 및 메모리 접근 시간을 단축합니다.

## [COMMON-050] 내부 단편화와 외부 단편화의 차이에 대해 설명해 주세요.

답변:
내부 단편화는 메모리를 관리하기 위해 나누어둔 공간단위를 프로세스가 사용하는 메모리 공간보다 커 낭비되는 문제를 의미합니다.
외부 단편화는 프로세스의 실행/종료가 반복되며 중간중간 생긴 미사용 메모리 영역들로 인해, 잔여 메모리 양의 총합은 충분하지만 실제로는 프로세스를 실행할 수 없는 문제를 의미합니다.

추가답변:

이를 완화하거나 해결하기 위해 페이징, 세그멘테이션, 메모리 풀 등의 기법을 사용할 수 있습니다.

페이징은 물리 메모리를 고정된 크기의 Frame으로 나누고 프로세스의 Page를 비연속적인 Frame에 할당할 수 있으므로 외부 단편화를 해결할 수 있습니다. 다만 마지막 Page에 사용하지 않는 공간이 남는 등의 내부 단편화는 발생할 수 있습니다.

세그멘테이션은 프로세스가 필요로 하는 논리적인 크기에 따라 가변 크기의 Segment를 할당하므로 내부 단편화를 줄일 수 있습니다. 다만 가변 크기의 메모리 할당과 해제가 반복되기 때문에 외부 단편화가 발생할 수 있습니다.

메모리 풀은 일정한 크기의 메모리 블록을 미리 할당해 두고 사용 및 반납하는 방식입니다. 이를 통해 메모리 할당 비용을 줄이고 외부 단편화를 일정 수준 통제할 수 있지만, 요청한 크기보다 큰 블록을 할당하면 내부 단편화가 발생할 수 있습니다.

## [COMMON-053] Critical Section이 무엇이고, 임계 구역 문제를 해결하기 위한 조건을 설명해 주세요.

답변:
Race Condition 문제를 해결하기 위해서는 공유자원에 대한 접근을 적절히 제한해야 합니다. 여러 스레드가 동시에 공유자원을 변경하지 못하도록 공유자원에 접근하는 코드의 진입과 진출을 제어할 수 있으며, 이때 공유자원에 접근하는 코드 영역을 Critical Section이라고 합니다.

Critical Section 문제를 해결하기 위해서는 크게 상호배제, 진행, 한정대기 조건이 충족되어야 합니다.

상호배제는 동일한 공유자원에 접근하는 Critical Section에는 동시에 하나의 프로세스 또는 스레드만 진입할 수 있어야 한다는 조건입니다.

진행은 Critical Section에 실행 중인 프로세스가 없고 진입을 원하는 프로세스가 존재한다면, Critical Section에 진입하지 않은 다른 프로세스가 해당 결정을 무한히 지연시켜서는 안 된다는 조건입니다.

마지막으로 한정대기는 어떤 프로세스가 Critical Section 진입을 요청한 이후, 다른 프로세스들이 먼저 진입할 수 있는 횟수에 제한이 있어야 한다는 조건입니다. 이를 통해 특정 프로세스가 무한히 대기하는 Starvation을 방지할 수 있습니다.

## [COMMON-054] Mutex와 Semaphore의 차이에 대해 설명해 주세요.

답변:
Mutex는 Critical Section에 진입할 수 있는 스레드의 개수가 하나일 때 사용하는 상호배제 구현 방식입니다. Critical Section에 접근할 때 Lock을 획득한 스레드만 임계영역에 접근할 수 있고, 다른 스레드들은 Lock이 해제될 때까지 대기합니다.
Semaphore는 내부의 Counter 값을 통해 동시에 접근할 수 있는 프로세스나 스레드의 개수를 제어하는 동기화 방식입니다. 스레드가 자원을 획득하면 Counter를 감소시키고, 자원을 반납하거나 신호를 보내면 Counter를 증가시킵니다.

Mutex와 Semaphore는 동시에 접근할 수 있는 스레드의 개수뿐만 아니라 소유권의 존재 여부에서 차이가 있습니다.
Mutex는 일반적으로 Lock을 획득한 스레드가 직접 해제해야 하지만, Semaphore는 특정 스레드가 소유한다는 개념이 없으므로 자원을 획득한 스레드가 아니더라도 Counter를 증가시켜 대기 중인 스레드를 깨울 수 있습니다.
따라서 Mutex는 주로 공유자원에 대한 상호배제를 위해 사용하고, Semaphore는 여러 개의 동일한 자원을 관리하거나 스레드 간 실행 순서 및 이벤트를 전달하는 신호 용도로 사용할 수 있습니다.

## [COMMON-055] Deadlock이 무엇이고, 발생 조건 4가지를 설명해 주세요.

답변:
Deadlock이란 둘 이상의 스레드가 서로가 점유하고 있는 자원이 필요해 서로 대기하게 되어 무한대기상태에 빠지는 현상을 의미합니다.
이 현상이 발생하기 위해서는 크게 4가지의 발생조건이 만족되어야 합니다.
우선 자원은 한 프로세스만 사용할 수 있도록 설정되어 있어야 하며, 다른 프로세스에 할당된 자원을 강제로 뺏을 수 없어야 합니다.
또한 최소 1개의 자원을 점유한 상태로 다른 스레드가 점유하고 있는 자원을 위해 대기하는 스레드가 존재해야 하고, 그때 다른 스레드는 앞서 언급한 스레드가 점유하고 있는 자원을 위해 대기함으로서 권한 대기를 위한 순환루프가 만들어져야 합니다.
이러한 요건을 순서대로 상호 배제, 점유 대기, 비선점, 순환 대기라고 하며, 이 4가지 조건이 모두 성립할 때 Deadlock이 발생합니다.

## [COMMON-056] Deadlock을 예방하거나 회피하는 방법에 대해 설명해 주세요.

답변:
Deadlock을 예방하기 위해서는 앞서 언급한 네 가지 발생조건 중 하나라도 성립하지 않도록 만들면 됩니다.

상호배제를 없애고자 한다면 여러 프로세스가 공유자원을 동시에 사용할 수 있도록 해야 합니다.
점유대기를 없애고자 한다면 프로세스가 실행되기 전 필요한 모든 자원을 한 번에 할당받도록 하거나, 현재 보유한 자원을 모두 반납한 후 새로운 자원을 요청하도록 할 수 있습니다.
비선점 조건을 없애고자 한다면 프로세스가 다른 자원을 요청했으나 즉시 할당받지 못한 경우, 현재 점유 중인 자원을 반납하도록 할 수 있습니다.
순환대기를 없애고자 한다면 자원에 고유한 순서를 부여하고, 모든 프로세스가 해당 순서에 따라서만 자원을 요청하도록 할 수 있습니다.

추가답변:
아울러 Deadlock 회피는 발생조건 자체를 제거하는 것이 아니라, 자원을 할당한 이후에도 시스템이 안전 상태를 유지할 수 있는지를 확인한 뒤 자원을 할당하는 방법입니다.
따라서 Banker’s Algorithm과 같은 방법을 통해 모든 프로세스가 정상적으로 종료될 수 있는 Safe Sequence가 존재하는지를 검사하므로서 문제를 해결할 수도 있습니다.

## [COMMON-058] Hash Table이 무엇이고, 해시 충돌을 해결하는 방법을 설명해 주세요.

답변:
Hash Table은 Hash Function을 통해 Key를 배열의 Index 또는 Bucket에 매핑하고, Key-Value 형태로 데이터를 저장하는 자료구조입니다.
내부적으로 배열 형태로 구성되고, Hash Function을 통해 계산된 위치에 데이터를 저장하기 때문에 평균적으로 O(1)의 시간복잡도로 데이터의 탐색, 삽입, 삭제를 수행할 수 있다는 장점이 있습니다.

해시함수의 출력범위는 유한하기 때문에 서로 다른 여러 입력에 동일한 출력값이 발생하는 Hash Collision이 발생할 수 있으므로, 이를 해결하기 위해 분리 연결법과 개방 주소법을 사용합니다.
분리 연결법은 동일한 Bucket에 저장되는 여러 데이터를 Linked List나 Tree 등의 구조로 연결하여 관리하는 방법입니다.
개방 주소법은 충돌이 발생했을 때 Hash Table 내부의 다른 빈 위치를 찾아 데이터를 저장하는 방법입니다. 대표적으로 일정한 간격으로 다음 위치를 탐색하는 Linear Probing, 탐색 간격을 제곱 단위로 증가시키는 Quadratic Probing, 별도의 Hash Function으로 탐색 간격을 계산하는 Double Hashing 등이 있습니다.

## [COMMON-059] Stack, Queue, Deque의 차이와 사용 사례를 설명해 주세요.

답변:
Stack은 후입선출방식의 자료구조로서, push 연산을 통해 삽입, pop 연산을 통해 삭제를 수행합니다. 입구와 출구가 동일하며, 이를 top이라 부릅니다. 보통 DFS 구현이나 함수호출 관리 등에 본 자료구조를 활용합니다.
Queue는 선입선출방식의 자료구조로서, enqueue 연산을 통해 삽입, dequeue 연산을 통해 삭제를 수행합니다. 입구와 출구가 서로 다르고, 입구를 rear, 출구를 front라고 부릅니다. 일반적으로 BFS 구현, 또는 어떤 작업에 대한 순차대기열을 구현 시 해당 자료구조가 많이 사용됩니다.
Deque는 Stack과 Queue를 합친 자료구조로서, 입력과 출력을 수행할 수 있는 데이터 입출구가 2개인 자료구조입니다. 따라서 데이터 삽입은 append/appendLeft, 데이터 삭제는 pop/popLeft로 각각 2개씩 존재합니다. Sliding Window 기법의 알고리즘 구현이나 회문검사 코드 등을 작성 시 활용됩니다.

## [COMMON-060] Tree, Binary Tree, Binary Search Tree의 차이에 대해 설명해 주세요.

답변:
Tree는 Node와 Edge로 구성되며, Cycle이 없는 계층적인 자료구조입니다. 자료구조에서 사용하는 Rooted Tree는 하나의 Root Node를 시작으로 여러 Child Node가 하위 계층으로 연결되는 구조를 의미합니다.

Binary Tree는 각 Node가 최대 두 개의 Child Node를 가질 수 있는 Tree를 의미합니다. 따라서 각 Node는 0개, 1개 또는 2개의 Child Node를 가질 수 있습니다.

Binary Search Tree는 Binary Tree에 데이터의 순서 조건을 추가한 자료구조입니다. 일반적으로 현재 Node보다 작은 값은 왼쪽 Subtree에 저장되고, 현재 Node보다 큰 값은 오른쪽 Subtree에 저장됩니다. 각 Subtree도 동일한 조건을 만족해야 합니다. 이를 통해 특정 값을 탐색할 때 비교 결과에 따라 왼쪽 또는 오른쪽 Subtree만 탐색할 수 있습니다.
