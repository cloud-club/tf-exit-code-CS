## [COMMON-041] 프로세스 주소 공간이 어떻게 구성되는지 설명해 주세요.

답변:

프로세스 주소 공간은 운영체제가 각 프로세스에 제공하는 독립적인 가상 주소 공간입니다.

실행할 명령어가 저장되는 Code 영역, 전역 변수와 정적 변수가 저장되는 Data 영역, 동적으로 할당한 메모리를 관리하는 Heap 영역, 함수 호출과 지역 변수를 관리하는 Stack 영역으로 구성됩니다.

필요에 따라 공유 라이브러리나 메모리 매핑 파일이 배치되는 영역도 존재합니다. 각 프로세스는 서로 독립된 주소 공간을 사용하기 때문에, 같은 가상 주소를 쓰더라도 서로 다른 물리 메모리에 연결될 수 있습니다.

참고 자료:

## [COMMON-042] Code, Data, Heap, Stack 영역의 역할을 설명해 주세요.

답변:

Code 영역은 CPU가 실행할 명령어가 저장되는 영역입니다. 보통 읽기 전용으로 설정되며, 같은 프로그램을 실행하는 여러 프로세스가 공유하기도 합니다.

Data 영역에는 전역 변수와 정적 변수가 저장됩니다. 프로그램이 시작될 때 생성되어 종료될 때까지 유지됩니다.

Heap 영역은 프로그램 실행 중 malloc이나 new 등을 통해 동적으로 할당한 메모리가 저장되는 영역입니다. 필요한 시점에 할당하고 더 이상 쓰지 않으면 해제해야 합니다.

Stack 영역은 함수 호출 정보, 매개변수, 지역 변수, 반환 주소 등을 저장합니다. 함수가 호출될 때 생성되고 함수가 끝나면 자동으로 정리되며, 각 스레드는 별도의 Stack 영역을 가집니다.

## [COMMON-043] Stack과 Heap의 차이에 대해 설명해 주세요.

답변:

Stack은 함수 호출과 관련된 지역 변수, 매개변수, 반환 주소 등을 저장하는 메모리 영역입니다.

함수가 호출되고 종료되는 순서에 따라 자동으로 할당되고 해제되기 때문에 관리가 단순하고 접근 속도가 빠른 편입니다. 다만 사용할 수 있는 크기가 제한되어 있고, 각 스레드가 별도의 Stack을 가집니다.

Heap은 프로그램 실행 중 필요한 크기의 메모리를 동적으로 할당하기 위한 영역입니다. Stack보다 큰 데이터를 저장하거나, 함수가 끝난 뒤에도 데이터를 유지해야 할 때 사용합니다.

Heap 메모리는 직접 해제해야 하는데, 제대로 관리하지 않으면 Memory Leak이나 메모리 단편화로 이어질 수 있습니다.

## [COMMON-044] Stack Overflow가 발생하는 이유를 설명해 주세요.

답변:

Stack Overflow는 프로그램에 할당된 Stack 영역의 크기를 초과했을 때 발생합니다.

가장 흔한 원인은 종료 조건이 잘못된 재귀 함수가 무한히 호출되거나, 재귀 호출의 깊이가 지나치게 깊어지는 경우입니다. 함수가 호출될 때마다 매개변수, 지역 변수, 반환 주소를 담은 Stack Frame이 계속 쌓이기 때문입니다.

함수 내부에 크기가 매우 큰 지역 배열이나 변수를 선언한 경우에도 Stack 공간을 빠르게 소모해 Stack Overflow가 발생할 수 있습니다.

운영체제는 Stack 경계를 벗어난 접근을 감지하면 예외를 발생시키고 프로그램을 종료합니다.

참고 자료:
- https://velog.io/@enamu/스택오버플로우Stack-Overflow

## [COMMON-045] Memory Leak이 무엇이고, 어떤 상황에서 발생할 수 있는지 설명해 주세요.

답변:

Memory Leak은 프로그램이 더 이상 사용하지 않는 메모리를 해제하지 않아서 해당 메모리를 계속 점유하는 현상입니다.

C나 C++에서는 malloc이나 new로 할당한 메모리를 free나 delete로 해제하지 않았을 때 발생합니다.

Garbage Collection을 사용하는 언어라고 해서 안전한 것은 아닙니다. 더 이상 필요 없는 객체를 전역 변수, 컬렉션, 캐시 등이 계속 참조하고 있으면 그 객체가 회수되지 않아 Memory Leak과 비슷한 문제가 생깁니다.

Memory Leak이 쌓이면 프로그램의 메모리 사용량이 계속 늘어나고, 결국 성능 저하나 Out of Memory로 인한 종료로 이어집니다.

## [COMMON-046] 가상 메모리가 무엇이고, 왜 필요한지 설명해 주세요.

답변:

가상 메모리는 각 프로세스에게 독립적이고 연속된 메모리 공간이 있는 것처럼 제공하는 메모리 관리 방식입니다.

프로세스가 사용하는 가상 주소는 MMU와 페이지 테이블을 거쳐 실제 물리 메모리 주소로 변환됩니다.

가상 메모리를 사용하면 프로세스마다 주소 공간을 분리할 수 있어, 다른 프로세스의 메모리에 함부로 접근하는 것을 막을 수 있습니다. 또한 프로그램을 물리 메모리의 연속된 위치에 배치하지 않아도 되므로 메모리를 유연하게 쓸 수 있습니다.

지금 당장 필요한 페이지만 물리 메모리에 올리고 나머지는 디스크에 두는 Demand Paging을 쓰면, 물리 메모리보다 큰 주소 공간도 사용할 수 있습니다.

참고 자료:
- https://velog.io/@redgem92/운영체제-가상-메모리

## [COMMON-047] 페이징이 무엇이고, 페이지와 프레임의 차이를 설명해 주세요.

답변:

페이징은 가상 메모리와 물리 메모리를 같은 크기의 고정된 블록으로 나누어 관리하는 방식입니다.

가상 주소 공간에서 나눈 블록을 Page라고 하고, 물리 메모리에서 나눈 블록을 Frame이라고 합니다.

운영체제는 페이지 테이블로 각 Page가 어떤 Frame에 저장되어 있는지 관리합니다. Page와 Frame의 크기가 같기 때문에, 하나의 Page를 물리 메모리의 비어 있는 아무 Frame에나 배치할 수 있습니다.

덕분에 프로세스의 메모리를 물리 메모리의 연속된 공간에 저장할 필요가 없어지고, 외부 단편화 문제도 크게 줄어듭니다.

참고 자료:
- https://velog.io/@seorim0801/프레임과-페이지

## [COMMON-048] Page Fault가 발생했을 때 처리 과정을 설명해 주세요.

답변:

Page Fault는 프로세스가 접근하려는 Page가 현재 물리 메모리에 없을 때 발생합니다.

Page Fault가 생기면 CPU는 실행을 잠시 멈추고 운영체제의 Page Fault Handler로 제어를 넘깁니다.

운영체제는 먼저 해당 메모리 접근이 유효한지 확인합니다. 유효하지 않은 주소라면 Segmentation Fault 같은 오류를 발생시키고, 유효한 접근이라면 디스크에서 필요한 Page를 찾습니다.

그다음 빈 Frame을 확보해 Page를 물리 메모리에 적재합니다. 빈 Frame이 없으면 페이지 교체 알고리즘으로 기존 Page 중 하나를 골라 내보냅니다.

마지막으로 페이지 테이블을 갱신하고, 멈췄던 명령어를 다시 실행합니다.

참고 자료:
- https://velog.io/@orcasuit/Page-Fault

## [COMMON-049] TLB가 무엇이고, TLB를 사용하면 왜 주소 변환이 빨라지는지 설명해 주세요.

답변:

TLB는 Translation Lookaside Buffer의 약자로, 최근에 사용한 가상 주소와 물리 주소의 변환 결과를 저장하는 CPU 내부의 캐시입니다.

가상 주소를 물리 주소로 바꾸려면 페이지 테이블을 조회해야 하는데, 페이지 테이블은 메모리에 있기 때문에 변환할 때마다 추가적인 메모리 접근이 필요합니다.

TLB에 필요한 변환 정보가 있는 TLB Hit가 발생하면, 페이지 테이블을 거치지 않고 바로 물리 주소를 얻을 수 있어 주소 변환이 빨라집니다.

반대로 TLB Miss가 발생하면 페이지 테이블을 조회한 뒤 그 결과를 TLB에 저장해 둡니다.

참고 자료:
- https://velog.io/@ttc1018/Translation-Lookaside-Buffer

## [COMMON-050] 내부 단편화와 외부 단편화의 차이에 대해 설명해 주세요.

답변:

내부 단편화는 할당된 메모리 블록 안에 쓰이지 않는 공간이 남는 현상입니다.

Page 크기가 4KB인데 프로세스가 마지막 Page에서 1KB만 쓴다면, 나머지 3KB는 다른 용도로 쓸 수 없어 내부 단편화가 됩니다.

외부 단편화는 남은 전체 메모리 크기는 충분하지만, 빈 공간이 여기저기 나뉘어 있어서 필요한 크기의 연속된 메모리를 할당하지 못하는 현상입니다.

고정된 크기의 Page와 Frame을 쓰는 페이징은 외부 단편화를 없앨 수 있지만, 마지막 Page 안에서 남는 공간 때문에 내부 단편화는 생길 수 있습니다.

## [COMMON-051] 동시성과 병렬성의 차이에 대해 설명해 주세요.

답변:

동시성은 여러 작업이 같은 시간대에 번갈아 실행되면서 함께 진행되는 성질을 의미합니다.

예를 들어 코어가 하나뿐이어도 운영체제가 작업을 빠르게 전환하면, 한 순간에는 하나씩 실행되더라도 사용자에게는 여러 작업이 동시에 돌아가는 것처럼 보입니다. 컨텍스트 스위칭으로 번갈아 실행하는 것이지 실제로 동시에 실행되는 것은 아닙니다.

반면 병렬성은 여러 CPU 코어나 처리 장치를 사용해 둘 이상의 작업을 실제로 같은 순간에 실행하는 것을 의미합니다. 코어가 4개라면 4개의 작업이 물리적으로 동시에 수행될 수 있습니다.

정리하면 동시성은 여러 작업을 어떻게 나눠서 다룰지에 대한 문제에 가깝고, 병렬성은 여러 작업을 물리적으로 동시에 실행해 처리 속도를 높이는 것에 가깝습니다.

참고 자료:
- https://velog.io/@cha-suyeon/멀티태스킹1-동시성-병렬성Concurrency-Parallelism

## [COMMON-052] Race Condition이 무엇이고, 어떤 상황에서 발생하는지 설명해 주세요.

답변:

Race Condition은 여러 스레드나 프로세스가 공유 자원에 동시에 접근할 때, 실행 순서에 따라 결과가 달라지는 문제입니다.

예를 들어 두 스레드가 하나의 공유 변수 값을 읽고 각각 1씩 증가시킨 뒤 저장한다고 하면, 두 연산이 겹치면서 증가 결과 중 하나가 사라질 수 있습니다.

이런 문제는 공유 데이터를 바꾸는 연산이 원자적으로 수행되지 않고, 접근 순서도 제대로 제어되지 않을 때 발생합니다.

Race Condition을 막으려면 Mutex, Semaphore, Atomic 연산 같은 동기화 방법을 사용합니다.

참고 자료:
- https://velog.io/@klloo/운영체제-경쟁-상태-Race-Condition

## [COMMON-053] Critical Section이 무엇이고, 임계 구역 문제를 해결하기 위한 조건을 설명해 주세요.

답변:

Critical Section은 여러 스레드나 프로세스가 공유 자원에 접근해서 값을 읽거나 바꾸는 코드 영역입니다.

여러 실행 흐름이 Critical Section에 동시에 들어가면 Race Condition이 생길 수 있어 동기화가 필요합니다.

임계 구역 문제를 풀려면 세 가지 조건을 만족해야 합니다.

첫 번째는 상호 배제입니다. 한 실행 흐름이 Critical Section을 실행하는 동안 다른 실행 흐름은 진입할 수 없어야 합니다.

두 번째는 진행입니다. Critical Section이 비어 있다면, 진입하려는 실행 흐름 중 하나가 불필요하게 지연되지 않고 들어갈 수 있어야 합니다.

세 번째는 한정 대기입니다. 특정 실행 흐름이 Critical Section에 진입하려고 무한히 기다리는 Starvation이 생기지 않아야 합니다.

참고 자료:
- https://velog.io/@chappi/OS는-할껀데-핵심만-합니다.-8편-Critical-section-임계-구역

## [COMMON-054] Mutex와 Semaphore의 차이에 대해 설명해 주세요.

답변:

Mutex와 Semaphore는 여러 스레드나 프로세스가 공유 자원에 동시에 접근하는 것을 제어하는 동기화 도구입니다.

Mutex는 하나의 공유 자원에 한 번에 하나의 실행 흐름만 접근하도록 잠그는 방식입니다. Mutex를 획득한 스레드가 작업을 마친 뒤 직접 잠금을 풀어야 하기 때문에 소유권이라는 개념이 있습니다.

Semaphore는 내부의 정수 값으로 동시에 접근할 수 있는 실행 흐름의 수를 제한합니다. 값이 1인 Binary Semaphore는 Mutex와 비슷하게 동작하고, 값이 여러 개인 Counting Semaphore는 제한된 개수의 자원을 관리할 때 씁니다.

그래서 Mutex는 하나의 Critical Section을 보호할 때 주로 쓰고, Semaphore는 일정 개수의 연결이나 자원의 동시 접근을 제한할 때 씁니다.

참고 자료:
- https://velog.io/@heetaeheo/뮤텍스Mutex와-세마포어Semaphore의-차이

## [COMMON-055] Deadlock이 무엇이고, 발생 조건 4가지를 설명해 주세요.

답변:

Deadlock은 둘 이상의 프로세스나 스레드가 서로 상대방이 가진 자원을 기다리면서, 어느 쪽도 작업을 진행하지 못하는 상태입니다.

Deadlock이 발생하려면 네 가지 조건이 동시에 성립해야 합니다.

첫 번째는 상호 배제입니다. 하나의 자원은 한 번에 하나의 실행 흐름만 사용할 수 있어야 합니다.

두 번째는 점유와 대기입니다. 이미 자원을 가진 상태에서 다른 자원을 추가로 기다려야 합니다.

세 번째는 비선점입니다. 다른 실행 흐름이 사용 중인 자원을 강제로 빼앗을 수 없어야 합니다.

네 번째는 순환 대기입니다. 여러 실행 흐름이 원형으로 연결되어 서로 다음 자원을 기다리는 상태가 만들어져야 합니다.

## [COMMON-056] Deadlock을 예방하거나 회피하는 방법에 대해 설명해 주세요.

답변:

Deadlock 예방은 발생 조건 네 가지 중 하나 이상이 성립하지 못하게 설계하는 방법입니다.

예를 들어 필요한 자원을 한 번에 모두 요청하게 하면 점유와 대기 조건을 없앨 수 있습니다. 모든 자원에 획득 순서를 정해 두고 같은 순서로만 Lock을 잡게 하면 순환 대기를 막을 수 있습니다.

Deadlock 회피는 자원을 할당하기 전에, 할당한 이후에도 시스템이 모든 작업을 정상적으로 끝낼 수 있는 안전 상태인지 확인하는 방식입니다. 대표적인 예가 은행원 알고리즘입니다.

이 밖에 Deadlock을 일단 허용해 두고 주기적으로 탐지한 뒤, 문제가 된 프로세스를 종료하거나 자원을 회수해서 복구하는 방법도 있습니다.

## [COMMON-057] Thread Safe하다는 것은 어떤 의미인지 설명해 주세요.

답변:

Thread Safe하다는 것은 여러 스레드가 같은 함수나 객체에 동시에 접근하더라도 데이터가 손상되지 않고 의도한 결과를 유지하는 것을 의미합니다.

공유 데이터를 아무 제어 없이 바꾸면 Race Condition이 생길 수 있어서, Thread Safe한 코드는 공유 자원 접근을 적절히 동기화해야 합니다.

Mutex나 Semaphore로 Critical Section을 보호하거나, Atomic 연산과 Thread Safe한 자료구조를 쓰는 방법이 대표적입니다.

이 밖에 값을 바꿀 수 없는 Immutable 객체나 각 스레드만 쓰는 지역 변수를 활용해서 공유 상태 자체를 줄이는 방법도 있습니다.

## [COMMON-058] Hash Table이 무엇이고, 해시 충돌을 해결하는 방법을 설명해 주세요.

답변:

Hash Table은 Key를 Hash Function에 넣어 얻은 값을 기준으로 데이터를 저장하고 조회하는 자료구조입니다.

Hash Function이 Key를 배열의 Index로 바꿔 주기 때문에, 평균적으로 데이터의 삽입, 검색, 삭제를 빠르게 처리할 수 있습니다.

서로 다른 Key가 같은 Index로 변환되는 현상을 해시 충돌이라고 합니다.

충돌을 푸는 방법으로는 같은 Index에 연결 리스트나 다른 자료구조를 두어 여러 데이터를 저장하는 Separate Chaining이 있습니다.

또 다른 방법은 충돌이 나면 배열의 다른 빈 자리를 찾는 Open Addressing입니다. Open Addressing에는 Linear Probing, Quadratic Probing, Double Hashing 등이 있습니다.

## [COMMON-059] Stack, Queue, Deque의 차이와 사용 사례를 설명해 주세요.

답변:

Stack은 마지막에 들어온 데이터가 먼저 나가는 LIFO 방식의 자료구조입니다. 함수 호출 관리, 실행 취소 기능, 괄호 검사, DFS 등에 사용합니다.

Queue는 먼저 들어온 데이터가 먼저 나가는 FIFO 방식의 자료구조입니다. 작업 대기열, 메시지 큐, 프린터 요청 처리, BFS 등에 사용합니다.

Deque는 Double-Ended Queue의 약자로, 앞과 뒤 양쪽에서 데이터를 넣고 뺄 수 있는 자료구조입니다.

Deque는 Stack과 Queue의 동작을 모두 구현할 수 있어서, 슬라이딩 윈도우나 양방향 탐색처럼 양쪽 끝 데이터를 함께 다뤄야 할 때 유용합니다.

## [COMMON-060] Tree, Binary Tree, Binary Search Tree의 차이에 대해 설명해 주세요.

답변:

Tree는 Node와 Edge로 구성되며, 부모와 자식 관계를 통해 계층적인 데이터를 표현하는 자료구조입니다.

Binary Tree는 각 Node가 최대 두 개의 자식 Node를 가질 수 있는 Tree입니다. 자식 Node는 보통 왼쪽과 오른쪽으로 구분합니다.

Binary Search Tree는 Binary Tree에 데이터 정렬 규칙을 더한 자료구조입니다. 한 Node를 기준으로 왼쪽 Subtree에는 더 작은 값이, 오른쪽 Subtree에는 더 큰 값이 저장됩니다.

이 규칙 덕분에 평균적으로 데이터를 빠르게 검색하고 삽입할 수 있습니다. 다만 Tree가 한쪽으로 치우치면 연결 리스트와 비슷한 구조가 되어 검색 성능이 떨어집니다.
