## [COMMON-001] CPU의 주요 구성 요소인 ALU, 제어장치, 레지스터에 대해 설명해 주세요.

답변:
컴퓨터는 핵심 부품은 크게 CPU, 메모리 , 보조기억장치, 입출력 장치로 나눌 수 있는데,
그 중 CPU는 ALU, 제어장치, 레지스터로 구성되어있다 
ALU는 산술 논리 연산 장치로 연산 자체를 담당하며  
레지스터는 CPU 내부의 작은 저장 장치로 레지스터에 명령어와 데이터를 저장해서 실제 연산에 재료로 사용한다
제어장치는 제어 신호를 내보내고 명령어를 해석하는 장치인데 메모리 읽기 신호와 메모리 쓰기 신호로 메모리 관련 신호를 보내는 역할을 한다 

1번지의 명령어(더하라, 3번지와 4번지)를 실행하고싶다면

1. 1번지에 있는 명령어를 읽어들여야하니까 1번지 메모리 읽기 제어 신호를 씀! 

2. 그 명령어를 CPU내부의 작은 저장장치인 레지스터에 저장함

3. 이 명령어를 제어 장치를 통해 해석함 (아, 3번지와 4번지를 더하라는거구나) 

4. 3번지와 4번지의 데이터를 메모리에서 읽어옴 by 제어신호 

5. 3번지 4번지를 레지스터에 저장함

6. ALU가 계산을 하고 그 결과를 레지스터에 저장함

## [COMMON-002] 명령어 사이클이 무엇인지 Fetch, Decode, Execute 단계로 설명해 주세요.

답변:
명령어가 해석되는 과정을 설명하면 
제어 장치에서 메모리 읽기 신호를 통해 명령어를 읽고 (Fetch) 
명령어를 제어 장치를 통해서 명령어를 해석하고 (Decode) 
이 명령어를 ALU가 연산한다 (Execute) 
이를 하나의 사이클이라고 볼 수 있다. 

특수 목적 레지스터 
PC : 다음 명령어 주소 
MAR : 메모리 주소 
MDR : 메모리 속 데이터
IR : 현재 명령어 저장

각 단계를 좀 더 자세히 살펴보면 PC에 저장되어있던 주소를 MAR(메모리 읽는 레지스터)로 보내고 
그 안에 들어있는 데이터를 MDR 레지스터로 저장하고 이 값이 다시 IR로 전송된다
해독 단계에서는 IR에 적재된 opcode를 해석해서 각 피연산자들을 레지스터에 저장해두고 
실행 단계에서는 ALU 제어 신호를 통해 연산을 수행하고 결과를 레지스터에 저장한다

## [COMMON-003] 프로그램이 실행되기까지의 과정을 컴파일, 링킹, 로딩, 실행 흐름으로 설명해 주세요.

답변: 
프로그래머가 코드를 작성한 파일 (.c or .python or .java)을 컴파일 해서 어셈블리어 형태인 object file로 변환하고 
링킹을 통해서 실행 파일을 만드는데 이 링킹 과저에서는 오브젝트 파일들과 라이브러리 파일들을 하나로 묶어서 .exe 파일이 됨 (여러 .obj -> 하나의 .exe) 
이제 로딩 과정을 통해 디스크같이 보조기억장치에 있던걸 주기억장치인 메모리에 올리고 
메모리에 로드가 완료되면 CPU가 해당 프로그램을 실행한다.
로드된 실행 파일이 실행될 때까지 자세히 설명하면 운영체제가 프로그램의 시작 주소를 PC에 입력하고 CPU는 PC에 있는 메모리 주소 명령어를 가져와서 해석하고 실행하고 이걸 종료할때까지 계속 반복하며 실행한다 

[정리]
컴파일: 소스 코드 ➔ 미완성 기계어(오브젝트 파일)

링킹: 오브젝트 파일 + 라이브러리 ➔ 완성된 실행 파일 (.exe)

로딩: 디스크의 실행 파일 ➔ 메인 메모리(RAM)로 적재

실행: CPU가 명령어 사이클(Fetch-Decode-Execute)을 돌며 처리

## [COMMON-005] JIT 컴파일이 무엇이고, 어떤 장점과 단점이 있는지 설명해 주세요.

답변:

JIT는 이걸 섞어서 처음에는 한줄씩 편하게 실행하다가 반복되는 핵심 코드였다면 통째로 번역해서 기억해두는 것! 

장점 : 엄청 빨라지고 어디에서나 잘 돌아간다 
단점: 처음에 어떤게 자주 돌아가는건지 (어떤걸 미리 번역해둘지) 결정하는게 오래걸리고 메모리가 더 먹는다

## [COMMON-014] 인터럽트가 무엇이고, 인터럽트가 발생했을 때 CPU와 운영체제가 어떻게 처리하는지 설명해 주세요.

답변:
인터럽트란 중간에 가로채서 이것부터 먼저해줘! 라고 신호를 보내는거다 
이때 컨텍스트 스위칭을 위해서 PCB에 현재 프로세스 상태를 백업해두고 커널 모드로 전환해서
인터럽트 벡터를 조회해서 발생한 인터럽트 번호에 맞는 해결 주소를 찾음
ISR 실행 (interrupt service routine)해서 인터럽트 요청 해결 
이후에 다시 이전에 진행하던 프로세스로 돌아가서 실행을 재개함 


*인터럽트는 인터럽트 핸들러가 처리하는데 
어떤 시스템 콜인지에 대해서는 약속된 레지스터에서 시스템 콜 테이블에서 인덱스로 확인하고 처리한다 (시스템 콜 전에 미리 레지스터에 값 채워두고 넘김) 
시스템 테이블은 해당 함수의 시작 주소값을 가지고 있고 그 주소값을 통해 시스템 콜 함수를 실행한다 

"
시스템 콜이 호출되면 컴파일 단계에서

## [COMMON-016] 프로세스의 상태에는 어떤 것들이 있고, 각 상태가 어떤 의미인지 설명해 주세요.

답변:
생성 (New): 프로세스가 막 만들어져서 메모리에 올라가기 위해 OS의 승인을 기다리는 단계다.

준비 (Ready): CPU만 할당받으면 즉시 일할 수 있게 모든 준비를 마치고 대기하는 상태다. (CPU 스케줄러의 간택을 기다리는 중!)

실행 (Running): CPU 제어권을 얻어서 실제로 명령어를 레지스터에 올리고 마구 계산하고 있는 상태다.

대기 (Blocked / Waiting): 파일 읽기/쓰기 같은 I/O 작업이나 특정 이벤트가 끝날 때까지 CPU를 반납하고 잠시 멈춰있는 상태다. 이 대기 상태에서는 아무리 CPU가 비어있어도 바로 일을 시작할 수 없고, 작업이 끝나서 준비(Ready) 상태로 넘어가야만 다시 CPU를 받을 수 있다.

종료 (Terminated): 할 일을 다 끝마치고 사용하던 메모리와 자원을 운영체제에 싹 반납하는 단계다

* Ready ➔ (스케줄러의 선택) ➔ Running ➔ (I/O 요청 발생) ➔ Blocked ➔ (I/O 완료 인터럽트) ➔ Ready
