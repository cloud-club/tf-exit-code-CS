## [COMMON-001] CPU의 주요 구성 요소인 ALU, 제어장치, 레지스터에 대해 설명해 주세요.

답변:

CPU의 주요 구성 요소로는 ALU, 제어장치, 레지스터가 있습니다.

ALU는 덧셈과 뺄셈 같은 산술 연산과 AND, OR, 시프트 등의 논리 연산을 수행합니다. 제어장치는 메모리에서 가져온 명령어를 해석하고, 각 장치에 제어 신호를 보내 명령어가 올바른 순서로 실행되도록 합니다. 레지스터는 CPU가 명령어를 실행하는 동안 필요한 데이터, 연산 결과, 주소와 다음 명령어의 위치 등을 임시로 저장합니다.

참고 자료:
- https://blog.naver.com/kyh30098/223199043147

## [COMMON-002] 명령어 사이클이 무엇인지 Fetch, Decode, Execute 단계로 설명해 주세요.

답변:

먼저 Fetch 단계에서는 Program Counter가 가리키는 메모리 주소에서 실행할 명령어를 가져와 CPU 내부의 IR 레지스터에 저장합니다. 이후 Program Counter는 다음 명령어의 주소를 가리키도록 갱신됩니다.

Decode 단계에서는 제어장치가 IR에 저장된 명령어를 해석하여 어떤 작업을 수행해야 하는지 판단합니다.

마지막으로 Execute 단계에서는 해석한 내용에 따라 ALU가 연산을 수행하거나 메모리에 접근하며, 필요한 경우 결과를 레지스터나 메모리에 저장합니다.

참고 자료:
- https://pre-walkingdeveloper.tistory.com/181

## [COMMON-003] 프로그램이 실행되기까지의 과정을 컴파일, 링킹, 로딩, 실행 흐름으로 설명해 주세요.

답변:

컴파일 단계에서는 작성한 소스 코드가 기계어로 이루어진 오브젝트 파일로 변환됩니다.

이후 링킹 단계에서는 여러 오브젝트 파일과 필요한 라이브러리를 연결하여 하나의 실행 파일을 생성합니다.

로딩 단계에서는 운영체제의 로더가 실행 파일의 코드와 데이터 영역을 프로세스의 가상 메모리 공간에 배치하고, 스택 등 실행에 필요한 환경을 준비합니다.

마지막으로 실행 단계에서는 CPU가 프로그램의 시작 지점부터 명령어를 수행하며, 프로그램은 하나의 프로세스로 동작하게 됩니다.

- https://velog.io/@dryrain/CS-Compile-Link-Load

## [COMMON-004] 컴파일 언어와 인터프리터 언어의 차이에 대해 설명해 주세요.

답변:

컴파일 방식은 프로그램을 실행하기 전에 소스 코드를 기계어 또는 실행 가능한 형태로 변환하는 방식입니다.

미리 변환된 결과를 실행하기 때문에 일반적으로 실행 속도가 빠르지만, 코드를 수정하면 다시 컴파일해야 하며 운영체제나 CPU 환경에 맞는 실행 파일이 필요할 수 있습니다.

반면 인터프리터 방식은 실행 시점에 코드를 해석하며 수행하는 방식입니다. 코드를 수정하고 바로 실행하기 편리하지만, 실행 과정에서 해석이 필요하기 때문에 일반적으로 컴파일 방식보다 실행 속도가 느릴 수 있습니다.

- https://modulabs.co.kr/blog/interpreter-and-compiler

## [COMMON-005] JIT 컴파일이 무엇이고, 어떤 장점과 단점이 있는지 설명해 주세요.

답변:

JIT 컴파일은 일반적인 컴파일과 달리 프로그램 실행 중에 바이트코드와 같은 중간 코드를 기계어로 컴파일하는 방식입니다.

처음에는 코드를 해석하며 실행하다가 자주 실행되는 부분인 핫스팟이 발견되면, 해당 코드를 기계어로 컴파일해 저장하고 이후부터는 컴파일된 코드를 직접 실행합니다.

이를 통해 자주 실행되는 코드의 성능을 높일 수 있다는 장점이 있습니다. 반면 실행 초기에 코드를 분석하고 컴파일하는 시간이 필요하며, 런타임 컴파일 과정에서 CPU와 메모리를 추가로 사용한다는 단점이 있습니다.

참고 자료:
- https://velog.io/@jeongbeom4693/컴파일러-인터프리터-JIT-컴파일

## [COMMON-006] 프로그램과 프로세스의 차이에 대해 설명해 주세요.

답변:

프로그램은 디스크에 저장되어 있는 정적인 형태의 파일로, 실행에 필요한 명령어와 데이터의 집합입니다.

프로세스는 프로그램이 메모리에 적재되어 실행 중인 상태로, 운영체제로부터 CPU와 메모리 등의 자원을 할당받습니다. 또한 각 프로세스는 독립된 가상 주소 공간과 실행에 필요한 레지스터 상태 등의 정보를 가집니다.

## [COMMON-007] 프로세스와 스레드의 차이에 대해 설명해 주세요.

답변:

프로세스는 운영체제가 프로그램을 실행시켰을 때 자원이 할당되는 단위이고, 각 프로세스들은 서로 독립된 가상 주소 공간을 갖습니다.

스레드는 프로세스 내부에서 실제 실행되는 단위이며, CPU 스케줄링을 받는 단위입니다. 같은 프로세스에 속하는 스레드는 같은 주소 공간을 공유하며, 각 스레드들은 PC와 레지스터 상태를 갖습니다.

이러한 스레드의 특징은 여러 작업을 효율적으로 처리할 수 있도록 하지만, 같은 자원에 동시에 접근하며 발생하는 race condition 등의 문제도 존재합니다.

참고 자료:
- https://velog.io/@raejoonee/프로세스와-스레드의-차이

## [COMMON-008] PCB가 무엇이고, 운영체제가 PCB를 사용하는 이유를 설명해 주세요.

답변:

PCB는 운영체제가 각 프로세스를 관리하기 위해 커널 영역에 유지하는 자료구조입니다.

PCB에는 프로세스의 ID, 실행 상태, 레지스터 값, 스케줄링 정보 등이 저장되며, 다른 프로세스로의 컨텍스트 스위칭이 발생할 때 기존에 실행 중인 프로세스의 정보를 저장하고 다음에 실행할 프로세스의 정보를 복원하기 위해서 사용됩니다.

참고 자료:
- https://jwprogramming.tistory.com/16

## [COMMON-009] 컨텍스트 스위칭이 무엇이고, 언제 발생하는지 설명해 주세요.

답변:

컨텍스트 스위칭은 CPU가 현재 실행 중인 프로세스나 스레드의 상태를 저장하고, 다음에 실행할 대상의 상태를 복원하여 CPU를 사용할 대상을 교체하는 과정입니다.

주로 실행 중인 프로세스가 I/O 요청으로 대기 상태가 되거나, 타임 슬라이스가 만료되거나, 스케줄링 정책에 따라 다른 프로세스가 선택될 때 발생합니다.

프로세스 컨텍스트 스위칭에서는 현재 프로세스의 PC와 레지스터 값 등의 실행 문맥을 PCB에 저장하고, 다음 프로세스의 문맥을 복원합니다.

참고 자료:
- https://velog.io/@nnnyeong/OS-Context-Switching-PCB-Process-Control-Block

## [COMMON-010] 프로세스 컨텍스트 스위칭과 스레드 컨텍스트 스위칭의 차이에 대해 설명해 주세요.

답변:

프로세스 컨텍스트 스위칭과 스레드 컨텍스트 스위칭의 가장 큰 차이는 주소 공간을 전환하는지 여부입니다.

프로세스 간 컨텍스트 스위칭에서는 PC와 레지스터 값 등의 실행 문맥뿐만 아니라, 서로 다른 가상 주소 공간을 사용하기 때문에 페이지 테이블과 같은 메모리 문맥도 전환해야 합니다. 이 과정에서 TLB와 캐시의 효율이 낮아질 수 있어 상대적으로 전환 비용이 큽니다.

반면 같은 프로세스에 속한 스레드들은 주소 공간을 공유하므로 페이지 테이블을 전환할 필요가 없습니다. 주로 PC, 레지스터 값, 스택 포인터 등의 스레드별 실행 문맥을 교체하므로 일반적으로 프로세스 컨텍스트 스위칭보다 비용이 작습니다.

참고 자료:
- https://engineerinsight.tistory.com/284

## [COMMON-011] 운영체제의 유저 모드와 커널 모드를 구분하는 이유를 설명해 주세요.

답변:

유저 모드와 커널 모드를 구분하는 이유는 시스템의 보안과 안정성을 보호하기 위해서입니다.

모든 프로그램이 커널 메모리나 입출력 장치와 같은 하드웨어 자원에 직접 접근할 수 있다면, 하나의 프로그램 오류나 악의적인 동작이 다른 프로세스나 운영체제 전체에 영향을 줄 수 있습니다.

이를 방지하기 위해 일반 프로그램은 제한된 권한을 가진 유저 모드에서 실행되며, 하드웨어 제어나 메모리 관리처럼 높은 권한이 필요한 작업은 커널 모드에서만 수행할 수 있습니다. 사용자 프로그램은 이러한 기능이 필요할 때 시스템 콜을 통해 커널에 요청하고, 커널은 요청의 권한과 유효성을 확인한 뒤 작업을 대신 수행합니다.

참고 자료:
- https://zu-techlog.tistory.com/122

## [COMMON-012] 시스템 콜이 무엇이고, 사용자 프로그램이 시스템 콜을 호출하는 이유를 설명해 주세요.

답변:

시스템 콜은 유저 모드에서 동작하는 프로그램이 운영체제 커널에서 제공하는 기능을 사용하기 위해 호출하는 공식 인터페이스입니다.

유저 모드에서 실행되는 프로그램은 커널 권한이 필요한 작업을 직접 수행할 수 없습니다. 따라서 시스템 콜을 통해 커널에 작업을 요청하고, 커널이 해당 작업을 수행한 뒤 결과를 전달합니다.

개발자는 일반적으로 라이브러리에서 제공하는 함수 형태로 시스템 콜을 사용하며, 대표적인 예로 open, read, write, fork, exec 등이 있습니다. 일반 함수 호출과 달리 시스템 콜이 실행되면 유저 모드에서 커널 모드로 전환됩니다.

참고 자료:
- https://velog.io/@klloo/운영체제-시스템-콜

## [COMMON-013] 시스템 콜이 실행될 때 유저 모드에서 커널 모드로 전환되는 과정을 설명해 주세요.

답변:

사용자 프로그램에서 read()와 같은 함수를 호출하면 시스템 콜 번호와 필요한 인자가 준비됩니다.

이후 syscall과 같은 명령어가 실행되면 CPU는 유저 모드에서 커널 모드로 전환되고, 운영체제가 미리 지정한 시스템 콜 진입점으로 이동합니다.

커널은 전달받은 시스템 콜 번호에 해당하는 기능을 실행하고, 작업이 끝나면 결과와 기존 실행 상태를 복원하여 다시 유저 모드로 돌아갑니다.

이 과정에서 사용자 프로그램은 커널 내부의 기능을 직접 실행하는 것이 아니라, 정해진 시스템 콜 인터페이스를 통해 커널에 작업을 요청합니다.

참고 자료:
- https://velog.io/@cocoding/유저모드-커널모드-인터럽트-시스템-콜

## [COMMON-014] 인터럽트가 무엇이고, 인터럽트가 발생했을 때 CPU와 운영체제가 어떻게 처리하는지 설명해 주세요.

답변:

인터럽트는 입출력 완료, 타이머 만료와 같이 처리해야 할 사건이 발생했음을 CPU에 알리는 신호입니다.

인터럽트가 발생하면 CPU는 현재 실행 중인 명령어를 마친 뒤, 현재 PC와 레지스터 값 등의 실행 상태를 저장하고 커널 모드로 전환합니다.

이후 인터럽트 번호에 해당하는 인터럽트 핸들러를 실행하여 발생한 사건을 처리합니다. 처리가 완료되면 기존 실행 상태를 복원하여 원래 작업을 계속하거나, 스케줄링 결과에 따라 다른 프로세스로 컨텍스트 스위칭할 수 있습니다.

인터럽트를 사용하면 CPU가 입출력 장치의 상태를 계속 확인하지 않고, 실제 사건이 발생했을 때만 처리할 수 있습니다.

참고 자료:
- https://velog.io/@adam2/인터럽트

## [COMMON-015] 하드웨어 인터럽트와 소프트웨어 인터럽트의 차이에 대해 설명해 주세요.

답변:

하드웨어 인터럽트는 CPU 외부의 하드웨어 장치에서 발생하는 인터럽트입니다.

대표적으로 키보드 입력, 디스크 입출력 완료, 네트워크 패킷 도착, 타이머 만료 등이 있으며, 현재 실행 중인 명령어와 관계없이 발생할 수 있기 때문에 비동기적으로 동작합니다.

소프트웨어 인터럽트는 프로그램이 명령어를 실행하는 과정에서 발생합니다. 시스템 콜처럼 프로그램이 의도적으로 발생시키는 트랩이나, 0으로 나누기와 잘못된 메모리 접근처럼 명령어 실행 중 발생하는 예외가 여기에 포함될 수 있습니다.

두 방식 모두 CPU의 실행 상태를 저장한 뒤 커널의 핸들러로 제어가 이동하지만, 외부 장치에 의해 발생하는지 프로그램 실행 과정에서 발생하는지에 차이가 있습니다.

참고 자료:

## [COMMON-016] 프로세스의 상태에는 어떤 것들이 있고, 각 상태가 어떤 의미인지 설명해 주세요.

답변:

프로세스의 대표적인 상태에는 New, Ready, Running, Waiting, Terminated가 있습니다.

New는 프로세스가 생성되고 있는 상태이고, Ready는 실행 준비를 마친 뒤 CPU 할당을 기다리는 상태입니다.

Running은 CPU를 할당받아 실제로 명령어를 실행하고 있는 상태이며, Waiting은 입출력 완료나 특정 이벤트를 기다리는 상태입니다. Waiting 상태의 프로세스는 필요한 작업이 완료될 때까지 CPU를 할당받아도 실행을 이어갈 수 없습니다.

Terminated는 프로그램 실행이 끝나 프로세스가 종료되고 자원이 정리되는 상태입니다.

일반적으로 프로세스는 Ready 상태에서 스케줄링을 통해 Running 상태가 되며, 입출력을 요청하면 Waiting 상태로 이동합니다. 입출력이 완료되면 다시 Ready 상태로 이동하여 CPU 할당을 기다립니다.

참고 자료:
- https://velog.io/@narcoker/운영체제-프로세스-상태-new-ready-running-waiting-terminated

## [COMMON-017] CPU 스케줄러가 필요한 이유와 대표적인 스케줄링 알고리즘을 설명해 주세요.

답변:

CPU의 개수는 한정되어 있지만 실행해야 할 프로세스는 여러 개이기 때문에, 어떤 프로세스에 CPU를 할당할지 결정하는 CPU 스케줄러가 필요합니다.

CPU 스케줄러는 CPU 이용률과 처리량을 높이고, 프로세스의 대기 시간과 응답 시간을 줄이며 여러 프로세스가 공정하게 실행될 수 있도록 합니다.

대표적인 스케줄링 알고리즘으로는 먼저 도착한 프로세스부터 실행하는 FCFS, 실행 시간이 짧은 프로세스를 우선하는 SJF, 우선순위가 높은 프로세스를 먼저 실행하는 Priority Scheduling이 있습니다.

또한 각 프로세스에 일정한 시간을 할당하고 순서대로 실행하는 Round Robin과, 여러 우선순위 큐를 사용하여 프로세스를 이동시키는 Multilevel Feedback Queue가 있습니다.

각 알고리즘은 응답 시간, 공정성, 컨텍스트 스위칭 비용과 같은 측면에서 서로 다른 장단점을 가집니다.

- https://velog.io/@jonghne/CPU-스케줄러-개념과-스케줄링-알고리즘-종류

## [COMMON-018] 선점형 스케줄링과 비선점형 스케줄링의 차이에 대해 설명해 주세요.

답변:

비선점형 스케줄링은 한 번 CPU를 할당받은 프로세스가 작업을 종료하거나 입출력 대기 상태로 이동하는 등 스스로 CPU를 반환할 때까지 계속 실행하는 방식입니다.

컨텍스트 스위칭이 비교적 적고 구현이 단순하지만, 실행 시간이 긴 프로세스가 CPU를 점유하면 다른 프로세스의 대기 시간과 응답 시간이 길어질 수 있습니다. FCFS와 비선점형 SJF가 대표적인 예입니다.

선점형 스케줄링은 운영체제가 실행 중인 프로세스로부터 CPU를 강제로 회수할 수 있는 방식입니다.

타임 슬라이스가 만료되거나 더 높은 우선순위의 프로세스가 실행 준비 상태가 되면 현재 프로세스를 Ready 상태로 이동시키고 다른 프로세스를 실행합니다.

응답성이 좋다는 장점이 있지만 컨텍스트 스위칭 횟수가 증가하여 오버헤드가 발생할 수 있습니다. Round Robin과 SRTF 등이 대표적인 예입니다.

참고 자료:
- https://velog.io/@tyjk8997/비선점-스케줄링과-선점-스케줄링

## [COMMON-019] Round Robin 스케줄링에서 Time Quantum이 너무 크거나 작을 때 발생하는 trade-off를 설명해 주세요.

답변:

Round Robin은 각 프로세스에 Time Quantum만큼 CPU를 할당하고, 시간이 만료되면 해당 프로세스를 준비 큐의 뒤로 이동시키는 방식입니다.

Time Quantum이 너무 크면 대부분의 프로세스가 중간에 선점되지 않고 실행되기 때문에 FCFS와 유사하게 동작합니다. 컨텍스트 스위칭 횟수는 줄어들지만, 뒤에서 기다리는 프로세스의 응답 시간이 길어질 수 있습니다.

반대로 Time Quantum이 너무 작으면 여러 프로세스가 빠르게 CPU를 할당받기 때문에 응답성은 좋아질 수 있습니다. 하지만 컨텍스트 스위칭이 지나치게 자주 발생하면서 실행 상태를 저장하고 복원하는 오버헤드가 증가하고, 전체 처리량이 낮아질 수 있습니다.

따라서 Time Quantum은 컨텍스트 스위칭 비용을 과도하게 발생시키지 않으면서도 적절한 응답성을 제공할 수 있도록 설정해야 합니다.

- https://velog.io/@chappi/OS는-할껀데-핵심만-합니다.-6편-스케줄링3-선점형-스케줄링-알고리즘Round-Robin-SRT-우선-순위-방식-Multilevel-Queue

## [COMMON-020] 멀티태스킹이 가능한 이유를 CPU 스케줄링과 컨텍스트 스위칭 관점에서 설명해 주세요.

답변:

하나의 CPU 코어는 한 순간에 하나의 프로세스나 스레드만 실행할 수 있습니다.

그럼에도 여러 프로그램이 동시에 실행되는 것처럼 보이는 이유는 운영체제가 CPU 시간을 짧게 나누어 여러 프로세스에 번갈아 할당하기 때문입니다.

CPU 스케줄러는 타임 슬라이스 만료나 입출력 대기 등의 상황에서 다음에 실행할 프로세스를 선택합니다.

이후 컨텍스트 스위칭을 통해 현재 실행 중인 프로세스의 PC와 레지스터 값 등의 실행 상태를 PCB에 저장하고, 다음 프로세스의 실행 상태를 복원합니다.

이 과정이 매우 짧은 시간 동안 반복되기 때문에 사용자에게는 여러 프로그램이 동시에 실행되는 것처럼 보입니다. 이를 동시성이라고 하며, 멀티코어 환경에서는 여러 작업이 실제로 동시에 실행되는 병렬성도 가능합니다.

참고 자료:
- https://velog.io/@cha-suyeon/멀티태스킹1-동시성-병렬성Concurrency-Parallelism
